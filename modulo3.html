Dockerización Profesional del Entorno

1h 30min ▼
🎯 Objetivo

Crear imágenes Docker optimizadas para producción usando multi-stage builds, y orquestar servicios completos con Docker Compose.

⏱️ Bloque 1 · Por qué Docker en producción (15 min)
  • El problema del "en mi máquina funciona"
  • Reproducibilidad de entornos de forma garantizada
  • Aislamiento y seguridad entre servicios
  • Escalabilidad horizontal simplificada
⏱️ Bloque 2 · Multi-stage Dockerfile (25 min)

Principios de un Dockerfile profesional:

Ejemplo — Dockerfile multi-stage (Python)
# Stage 1: Builder
FROM python:3.12-slim AS builder
WORKDIR /app
COPY pyproject.toml .
RUN pip install --no-cache-dir build && \
    pip install --no-cache-dir .

# Stage 2: Production
FROM python:3.12-slim
WORKDIR /app
COPY --from=builder /usr/local/lib /usr/local/lib
COPY src/ ./src/

# Seguridad: usuario no-root
RUN adduser --disabled-password appuser
USER appuser

EXPOSE 8000
CMD ["python", "-m", "uvicorn", "src.main:app"]
Prompt — Optimización con IA
Revisá este Dockerfile y optimizalo para producción:
- Reducir tamaño de imagen al mínimo
- Eliminar capas innecesarias
- Aplicar mejores prácticas de seguridad
- Agregar health check adecuado
- Documentar cada decisión
⏱️ Bloque 3 · Docker Compose: orquestación completa (35 min)
Ejemplo — docker-compose.yml (App + DB + n8n)
version: '3.9'

services:
  app:
    build: .
    ports: ["8000:8000"]
    environment:
      DATABASE_URL: postgresql://user:pass@db:5432/myapp
    depends_on:
      db:
        condition: service_healthy
    restart: unless-stopped

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: myapp
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD", "pg_isready", "-U", "user"]
      interval: 10s
      retries: 5

  n8n:
    image: n8nio/n8n
    ports: ["5678:5678"]
    volumes:
      - n8ndata:/home/node/.n8n
    restart: unless-stopped

volumes:
  pgdata:
  n8ndata:
Prompt — Revisión de seguridad del Compose
Revisá este docker-compose.yml y sugerí mejoras de seguridad:
- Secrets management (sin passwords hardcodeadas)
- Redes aisladas entre servicios
- Límites de recursos (CPU/memoria)
- Volúmenes con permisos correctos
✅ Checklist de producción (5 min)
  • ¿La imagen no corre como root?
  • ¿Los secrets están en variables de entorno, no hardcodeados?
  • ¿Hay health checks definidos?
  • ¿El tamaño de imagen es razonable (< 200MB)?
📄 Entregables
  • Dockerfile multi-stage optimizado
  • docker-compose.yml con todos los servicios
  • docker-compose.override.yml para desarrollo local
  • .env.example documentado

Orquestación de Flujos de Mantenimiento en n8n

1h 30min ▼
🎯 Objetivo

Configurar n8n para automatizar tareas de monitoreo, backups y alertas sin escribir scripts manualmente.

⏱️ Bloque 1 · ¿Qué es n8n y por qué usarlo? (15 min)
  • Automatización visual de flujos de trabajo (workflows)
  • Open-source, self-hosted, sin vendor lock-in
  • Más de 400 integraciones nativas
  • Corre perfectamente dentro de Docker Compose
⏱️ Bloque 2 · Setup e instalación (15 min)
  • Con Docker (recomendado): imagen oficial n8nio/n8n
  • Sin Docker: npm install -g n8n && n8n start
  • Acceso en http://localhost:5678
  • Configuración de credenciales y variables de entorno
⏱️ Bloque 3 · Flujo: monitoreo de salud de DB (30 min)

Diseñar en n8n un workflow que:

  • Se ejecute cada 5 minutos (trigger de cron)
  • Conecte a la base de datos y ejecute SELECT 1
  • Si falla → envíe alerta por email / Slack / Telegram
  • Si el tiempo de respuesta supera 500ms → ejecute scripts de diagnóstico
  • Registre cada check en un log centralizado
Prompt — Diseño del flujo con IA
Diseñá un workflow de n8n para monitorear la salud de una base de datos PostgreSQL.

El flujo debe:
- Correr cada 5 minutos
- Verificar conectividad y tiempo de respuesta
- Enviar alerta a Slack si falla
- Ejecutar un script de limpieza si el storage supera el 80%
- Loguear todos los eventos en una tabla de auditoría
⏱️ Bloque 4 · Flujo: backup automatizado (25 min)
  • Trigger diario a las 03:00 AM
  • Ejecutar pg_dump via nodo Execute Command
  • Comprimir y nombrar con timestamp
  • Subir a S3 / Google Drive / almacenamiento local
  • Notificar resultado (éxito / fallo + tamaño del backup)
Prompt — Flujo de backup
Creá un workflow n8n que:
- Ejecute pg_dump cada día a las 3 AM
- Comprima el archivo resultante
- Lo suba a Google Drive con nombre: backup_YYYY-MM-DD.sql.gz
- Elimine backups con más de 30 días
- Envíe resumen por email
✅ Evaluación del módulo (5 min)
  • ¿El flujo de monitoreo detecta fallas reales?
  • ¿Los backups se generan y almacenan correctamente?
  • ¿Las alertas llegan al canal correcto?
📄 Entregables
  • Workflow n8n exportado: monitoring-flow.json
  • Workflow n8n exportado: backup-flow.json
  • Documentación de configuración de credenciales

Agentes de IA para CI/CD

1h 30min ▼
🎯 Objetivo

Conectar n8n con GitHub para automatizar el ciclo de vida completo del código, desde el push hasta el despliegue en producción.

⏱️ Bloque 1 · Integración n8n ↔ GitHub (20 min)
  • Configurar webhook en GitHub → n8n
  • Eventos disponibles: push, PR, issue, release, workflow_run
  • Autenticación con GitHub Personal Access Token
  • Trigger on: push a main → iniciar pipeline de deploy
Prompt — Diseño del pipeline con IA
Diseñá un pipeline de CI/CD usando n8n y GitHub webhooks que:

1. Se active con cada push a la rama main
2. Ejecute los tests automáticamente
3. Si los tests pasan: build de Docker image
4. Deploy al servidor de staging
5. Run smoke tests en staging
6. Si todo OK: deploy a producción
7. Notificar en Slack el resultado completo
⏱️ Bloque 2 · GitHub Actions + n8n combinados (30 min)
  • GitHub Actions para build y tests (CPU en los runners de GitHub)
  • n8n para orquestación de deploy y notificaciones (lógica de negocio propia)
  • Webhook de GitHub Actions hacia n8n para continuar el flujo post-CI
Prompt — GitHub Actions workflow
Generá un archivo .github/workflows/deploy.yml que:

- Corra en push a main
- Ejecute tests con matriz de versiones
- Construya imagen Docker
- La pushee a GitHub Container Registry
- Llame al webhook de n8n para continuar el deploy
⏱️ Bloque 3 · Rollback automático (30 min)
  • Monitoreo post-deploy de métricas (error rate, latencia)
  • Si métrica supera umbral → trigger automático de rollback
  • Rollback vía: docker service update --rollback o tag anterior
  • Notificación del incidente con causa y versión restaurada
Prompt — Flujo de rollback automático
Diseñá un flujo n8n que monitoree el estado post-deploy y,
si el error rate supera el 1% en 5 minutos:
- Ejecute rollback automático al tag anterior
- Abra un issue en GitHub con el reporte del incidente
- Notifique al equipo con contexto completo
📄 Entregables
  • .github/workflows/deploy.yml
  • Workflow n8n: cicd-orchestrator.json
  • Workflow n8n: auto-rollback.json
  • /docs/cicd-architecture.md

🐳 Resultado del Módulo 3

  • Entorno reproducible y seguro con Docker multi-stage en cualquier servidor
  • Monitoreo proactivo de base de datos con alertas y auto-reparación
  • Backups automatizados con verificación de integridad
  • Pipeline CI/CD completo: push → tests → deploy → rollback automático
