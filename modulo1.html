Módulo 1 · Semanas 1–2

🔍 Control Total y Auditoría

Levantá el proyecto, entendé la base de código existente y establecé el entorno de control. La IA actúa como lupa arquitectónica para mapear sistemas complejos en minutos.

1

Ingesta y Relevamiento de Proyectos Complejos

1h 30min ▼
🎯 Objetivo

Convertir un repositorio desconocido en un mapa mental y técnico claro en 90 minutos.

⏱️ Bloque 1 · Cambio de paradigma (10 min)
  • Antes: leer código manualmente, archivo por archivo
  • Ahora: hacer que la IA lea el sistema completo
  • Concepto clave: "No entendés archivo por archivo → entendés el sistema completo"
⏱️ Bloque 2 · Setup interactivo (15 min)
  • Cursor IDE con integración de IA
  • Gemini 1.5 Pro (ventana de contexto extendida)
  • Alternativa mobile: subir repo a GitHub + usar interfaz web + prompts
⏱️ Bloque 3 · Exploración total (25 min)

Los estudiantes ejecutan estos prompts sobre su repositorio:

Prompt 1 — Análisis general
Analiza este proyecto completo.

Quiero:
1. Qué hace el sistema
2. Módulos principales
3. Flujo de datos
4. Riesgos técnicos
Prompt 2 — Profundización
Identifica:
- Código duplicado
- Acoplamiento fuerte
- Posibles bugs
- Riesgos de seguridad
⏱️ Bloque 4 · Workshop (30 min)
  • Input: Repo real o ejemplo (Java / Python / Node)
  • Generar diagrama de arquitectura
  • Mapear flujo de datos
  • Identificar deuda técnica y vulnerabilidades
Output esperado
Crítico: vulnerabilidades de seguridad activas
Medio: deuda técnica y code smells
Bajo: mejoras de legibilidad y estructura
⏱️ Bloque 5 · Cierre y reflexión (10 min)
  • ¿Qué entendiste que antes te llevaba días?
  • ¿Qué NO entendió bien la IA? (limitaciones honestas)
📄 Entregable
  • Documento de arquitectura del sistema
  • Diagrama de flujo de datos
  • Lista priorizada de riesgos
2

Estrategias de Documentación Dinámica

1h 30min ▼
🎯 Objetivo

Crear documentación que no quede obsoleta, se genere automáticamente y sea útil tanto para humanos como para la IA.

⏱️ Bloque 1 · Concepto clave (15 min)

"La documentación ya no se escribe → se genera y se mantiene sola"

  • Documentación técnica: arquitectura, APIs, modelos de datos
  • Documentación funcional: flujos de negocio, casos de uso
  • Documentación operativa: runbooks, procedimientos de incidentes
⏱️ Bloque 2 · Setup (15 min)
  • Cursor con archivos de contexto
  • Markdown / Wiki integrado al repositorio
  • Archivos context.md como memoria del proyecto
⏱️ Bloque 3 · "Enseñarle a la IA" (25 min)

Crear el archivo /docs/context.md con las reglas de negocio:

Plantilla — context.md
Este sistema:
- Gestiona usuarios y autenticación
- Usa PostgreSQL como base de datos principal
- Expone una API REST (versión 2)

Reglas de negocio:
- No borrar datos críticos, solo soft-delete
- Validar todos los inputs antes de persistir
- Los emails son únicos por tenant
⏱️ Bloque 4 · Workshop (30 min)
Paso 1 — Generación inicial
Genera documentación completa del sistema:
- Arquitectura general
- Endpoints disponibles
- Modelo de datos y relaciones
Paso 2 — Auto-actualización
Cada vez que cambie este módulo,
actualiza la documentación en /docs
manteniendo el mismo formato.
Paso 3 — Versionado
  • Guardar docs en /docs/
  • Integrar con Git (commits automáticos de docs)
  • Desde mobile: editar Markdown directamente en GitHub
✅ Checklist de evaluación (5 min)
  • ¿Se entiende el sistema sin leer código?
  • ¿La IA puede usar esa documentación como contexto?
📄 Entregables
  • Wiki técnica completa en /docs/
  • Archivo de contexto reutilizable context.md
3

Gestión y Seguridad de la Base de Datos

1h 30min ▼
🎯 Objetivo

Entender y asegurar la base de datos usando IA como auditor especializado de esquemas y consultas.

⏱️ Bloque 1 · Introducción (15 min)

Problemas típicos en bases de datos heredadas:

  • Queries lentas sin índices adecuados
  • SQL injection por concatenación directa
  • Backups inexistentes o no testeados
  • Datos sensibles expuestos sin encriptación
⏱️ Bloque 2 · "Leer la DB como un mapa" (20 min)
Prompt — Análisis de esquema
Analiza este esquema de base de datos.

Quiero:
- Relaciones entre tablas
- Tablas y columnas críticas
- Problemas de diseño
- Índices faltantes
- Datos sensibles sin proteger
⏱️ Bloque 3 · Optimización con IA (20 min)

Ejemplo de query problemática:

Antes — vulnerable
SELECT * FROM users WHERE email = 'test'
Prompt — optimización
Optimiza esta query para performance y seguridad.
Explica cada mejora y por qué es importante.
Después — seguro y eficiente
cursor.execute(
    "SELECT id, name, email FROM users WHERE email = %s",
    (email,)
)
⏱️ Bloque 4 · Workshop completo (30 min)
  • Parte 1 — Auditoría: detectar SQL injection, falta de índices, datos sensibles expuestos
  • Parte 2 — Seguridad: encontrar vulnerabilidades en capa de acceso a datos
  • Parte 3 — Backup: diseñar estrategia de backups diarios con versionado y recuperación
Prompt — Auditoría completa
Encuentra vulnerabilidades en la capa de acceso a datos:
1. Inyecciones SQL posibles
2. Falta de validación de inputs
3. Permisos excesivos
4. Datos sin encriptar
✅ Buenas prácticas (5 min)
  • ORM con consultas parametrizadas (nunca concatenar)
  • Variables parametrizadas en queries directas
  • Logs de acceso a tablas sensibles
  • Principio de mínimo privilegio en usuarios de DB
📄 Entregables
  • Informe de seguridad de la base de datos
  • Queries optimizadas y parametrizadas
  • Estrategia documentada de backup automatizado

🧠 Resultado del Módulo 1

  • Leer cualquier base de código en minutos usando IA
  • Documentar sistemas automáticamente con contexto reutilizable
  • Auditar la seguridad básica de cualquier codebase
  • Entender bases de datos complejas y sus riesgos
  • Tener control real del sistema desde el primer día
