Estrategias de Despliegue y Rollback

1h 30min ▼
🎯 Objetivo

Configurar despliegues controlados con estrategias que minimicen el riesgo y permitan volver atrás en segundos si algo falla.

⏱️ Bloque 1 · Estrategias de despliegue (25 min)
Blue/Green

Dos entornos idénticos. Se cambia el router de Blue a Green. Rollback instantáneo cambiando el router de vuelta.

Canary Release

5% del tráfico va a la nueva versión. Si las métricas son buenas, se incrementa gradualmente hasta 100%.

Feature Flags

Deploy del código deshabilitado. Se activa por configuración para usuarios específicos sin hacer un nuevo deploy.

Rolling Update

Se reemplaza una instancia a la vez. Nunca hay downtime pero la transición tarda algunos minutos.

⏱️ Bloque 2 · Implementación con Docker (25 min)
Prompt — Estrategia de deploy blue/green
Diseñá una estrategia blue/green deployment para esta app dockerizada.

Incluí:
- Configuración del load balancer (Traefik / Nginx)
- Comando para hacer el switch de tráfico
- Validaciones automáticas antes de confirmar el switch
- Procedimiento de rollback (debe tardar menos de 60 segundos)
Comando — Rollback con Docker Swarm
# Rollback a la versión anterior del servicio
docker service update --rollback myapp_web

# Verificar estado
docker service ps myapp_web --filter "desired-state=running"
⏱️ Bloque 3 · Simulación de deploy fallido (30 min)

Ejercicio práctico de recuperación de incidente:

  • Simular un deploy con bug crítico en producción
  • Detectar el fallo a través de las métricas de monitoreo
  • Ejecutar rollback al tag anterior
  • Documentar el incidente (post-mortem breve)
  • Identificar qué tests hubieran prevenido el bug
Prompt — Diseño del sistema de métricas
Diseñá un sistema de monitoreo para detectar rápidamente un deploy fallido.

Métricas clave a vigilar:
- Error rate (HTTP 5xx)
- Latencia P95 y P99
- Tasa de requests procesadas
- Uso de CPU/memoria
- Health checks de la base de datos

Alertas: Slack inmediato si alguna métrica supera el umbral.
✅ Criterios de go/no-go (5 min)
Error rate < 0.1% en los primeros 5 minutos
Latencia P95 similar a la versión anterior (±20%)
Health checks de todos los servicios en verde
Si cualquier métrica supera umbral → rollback automático
📄 Entregables
  • /docs/deployment-strategy.md
  • /docs/runbook-rollback.md (guía paso a paso)
  • Dashboard de monitoreo configurado

Taller de Casos Reales (Mentoría)

1h 30min ▼
🎯 Objetivo

Trabajo intensivo sobre los proyectos específicos de cada estudiante, aplicando todo lo aprendido en los módulos anteriores a un contexto real.

⏱️ Formato de la clase (90 min)

Cada estudiante trabaja en su propio proyecto con soporte del docente y la IA:

  • Primeros 15 min: cada uno presenta el estado actual de su proyecto
  • 60 min: trabajo individual / en duplas con asistencia
  • 15 min: revisión conjunta de avances y bloqueos
🛠️ Casos de uso típicos a trabajar
  • Migración de base de datos PostgreSQL con datos reales
  • Dockerización de una app legacy que nunca tuvo contenedores
  • Agregar autenticación JWT a una API sin auth
  • Configurar CI/CD desde cero en un repo existente
  • Automatizar el deploy de un proyecto propio con n8n
  • Auditoría de seguridad de dependencias del proyecto actual
💡 Prompts de apoyo para la mentoría
Prompt — Diagnóstico de proyecto real
Analizá este proyecto que estoy migrando.

Tengo:
- Base de datos legacy en MySQL 5.7
- API en Flask 1.x sin tipado
- Sin tests, sin Docker, sin CI/CD

Generá un plan de modernización incremental de 4 semanas
que no corte el servicio en producción en ningún momento.
Prompt — Resolución de bloqueante
Estoy migrando de [tecnología A] a [tecnología B]
y me encuentro con este error:

[pegar el error completo]

Contexto del proyecto: [descripción breve]
¿Cuál es la causa raíz y cómo lo resuelvo?
📋 Conexión con el proyecto final

Esta clase es la antesala del proyecto final. Los avances de hoy se integrarán en la presentación de la Clase 12. Se espera que cada estudiante tenga al menos un componente funcionando end-to-end.

📄 Entregable
  • Avance documentado del proyecto personal
  • Al menos un entregable funcional (Docker, pipeline, tests o migración)
  • Lista de pendientes para el proyecto final

Presentación de Ecosistemas Autónomos

1h 30min ▼
🎯 Objetivo

Demostración del proyecto final: de código legado a una aplicación moderna, documentada, dockerizada y con mantenimiento automatizado. Cierre y reflexión del programa.

⏱️ Formato de presentación (60 min)
  • Cada estudiante tiene 10 minutos para presentar su proyecto
  • Demostración en vivo del sistema funcionando
  • 5 minutos de preguntas y feedback del grupo
📋 Criterios de evaluación del proyecto final
Arquitectura

¿Se entiende el sistema sin leer código? ¿Está documentado?

Docker

¿Levanta con un solo docker compose up? ¿Usa multi-stage?

Automatización

¿Hay al menos un flujo n8n funcionando (backup, monitoreo o deploy)?

Seguridad

¿Pasó una auditoría básica? ¿Sin secrets hardcodeados?

Testing

¿Existe al menos una suite de tests que corra en CI?

CI/CD

¿Hay un pipeline que haga deploy automático al pushear a main?

⏱️ Cierre del programa (30 min)
  • Reflexión: ¿Qué cambió en cómo ves los sistemas técnicos?
  • Próximos pasos: agentes autónomos, embeddings, pipelines de datos
  • Recursos: comunidad, bibliografía y proyectos open-source recomendados
  • Networking: conexiones y colaboraciones entre participantes
🔮 ¿Qué sigue después del programa?
  • Agentes autónomos que auditan y mejoran código sin intervención
  • Scraping + embeddings + Neo4j para grafos de conocimiento
  • Pipelines de ETL automatizados con IA
  • Dashboards de métricas conectados a los flujos de n8n
  • MLOps: despliegue y monitoreo de modelos en producción
📄 Entregable final
  • Repositorio público con toda la infraestructura funcionando
  • README.md profesional con instrucciones de setup
  • Video demo de 3 minutos del sistema corriendo
  • Documento de arquitectura completo en /docs/

🏆 Resultado del Programa Completo

  • Soberanía técnica: podés entrar en cualquier repo y entenderlo en minutos
  • Migración confiable con red de seguridad de tests automáticos
  • Infraestructura como código: Docker en producción sin "funciona en mi máquina"
  • Mantenimiento zero-touch: n8n gestiona logs, backups y alertas solo
  • Pipeline CI/CD completo con despliegue y rollback automáticos
